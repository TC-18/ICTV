Output for most scanners

Complex objects

Digital Curvature

There is a wide number of curvature estimators for digital objects

But a vast majority of them rely on parameters ...

  • Sphere radius
  • Neighbourhood window
  • ...

from which depends the quality of the result

Integral Invariants

Let a ball slide along the border of the object

Computes the curvature as a ratio between the ball volume and the object volume inside the ball

Is proven to have 3D convergency

Issue

Computing the curvature is a long process

Choosing the correct parameterization for an estimation is sometimes difficult

The solution: Interactive visualization

Pipeline

Our pipeline runs in two steps ...

  • Extracting a mesh from the digital object

  • Compute the curvature value on each vertex of this mesh

Both those steps are done on the fly and in real-time,
thanks to the GPU parallelism !

Real Time GPU Isosurface Extraction

Marching Cubes [LC87]

  • Grid division of the scene
  • Independent triangulation of each cell
  • Highly parallel


      

Marching Cubes

GPU loves Marching Cubes

Aligned with surfels

                                

Regular Grid

  • High number of triangles
  • Most of them behind from the camera or too far to be visible

Transvoxels [Len10]

Adaptation of Marching Cubes to use it with an octree instead of the regular grid

Uses a modified Marching Cubes algorithm to patch the cracks

Transvoxels [Len10]

Transition cells

Subdivision criteria

Based on a radial distance to the camera

Projected screen size at distance d

Radial distance measure

Quadtrees on the GPU [DIP14]

GPU really can't handle recursivity

Derecursifying the octree

Morton codes, each cell encode its path to the root

The whole tree is represented by a list of its leaves as uncorrelated nodes

          

Quadtrees on the GPU [DIP14]

Every time the camera moves, we update
the tree

  • The cell is kept
  • The cell is merged
  • The cell is subdivided

Pipeline

Curvature computation


$\VolC$ is obtained by counting the digital points $\vx /h $ that intersect $\textit{B}_{R/h}$.


A LOT of memory accesses

Reducing the number of texture accesses


The digital dataset is loaded on the GPU as a texture

Mipmaps !

Hierachical subdivision

Exact

Reduces the number of memory fetches

Not at all suited for the GPU

As slow as the naive version

Regular subdivision


$\VolC$ is obtained by summing the densities at a level $l$, $\vx_l /h_l$ that intersect $\textit{B}_{R/h}$.


Fewer memory accesses

Refinement over time

Keeping an exact curvature visualization while maintaining a real-time manipulation of the object

At each frame:

  1. If the object is moved { L=1 }

    Else { L=L-1 }

  2. Approximate curvature at level L
